# CLAUDE.md — правила разработки (Codenames Online)

Документ описывает обязательные правила для всех, кто вносит изменения в проект:
рабочий процесс с документацией, правила комментирования кода и архитектурные
принципы. Цель — сохранять код читаемым, предсказуемым и готовым к масштабированию.

> Связанные документы: каталог [docs/](docs/) — устройство приложения
> ([architecture.md](docs/architecture.md), [server.md](docs/server.md),
> [client.md](docs/client.md), [protocol.md](docs/protocol.md)) и запуск/доступ
> для игроков ([hosting.md](docs/hosting.md),
> [cloudflare-tunnel.md](docs/cloudflare-tunnel.md)).

---

## 0. Рабочий процесс с документацией (обязательно)

### 0.1. Перед началом работы — всегда смотреть в `docs`

Прежде чем писать или менять код, **сначала загляни в документацию** и найди там
информацию о приложении: как устроена архитектура, какие модули за что отвечают,
какие игровые правила и сетевые контракты уже зафиксированы.

- Стартовые точки: этот файл (`CLAUDE.md`), корневой [README.md](README.md) и
  каталог [docs/](docs/), где документация разбита по логике (архитектура,
  серверные/клиентские модули, контракт сообщений).
- Не начинай задачу «с нуля по памяти» — сверься с уже описанными решениями,
  чтобы не дублировать логику и не нарушать принятые контракты.
- Если в документации нет нужной информации — это сигнал, что её нужно дополнить
  (см. 0.3), а не действовать вслепую.

**Чтение docs — это жёсткий первый шаг, а не пожелание.** До того как открыть или
изменить хоть один файл с кодом, обязательно прочитать релевантные документы.
«Релевантные» определяются по типу правки:

| Тип правки | Что прочитать **до** кода |
|---|---|
| Любая | `CLAUDE.md` (этот файл) + [docs/architecture.md](docs/architecture.md) |
| Клиент (UI, рендер, DOM, события) | [docs/client.md](docs/client.md) |
| Сервер (логика, сервисы, таймеры, комнаты) | [docs/server.md](docs/server.md) |
| Новый/изменённый тип WebSocket-сообщения | [docs/protocol.md](docs/protocol.md) |

Порядок действий по задаче строго такой: **(1) прочитать docs по таблице → (2)
менять код → (3) обновить docs (см. 0.3).** Шаг 1 нельзя пропускать, даже если
правка выглядит как «одна строчка» или «просто стили»: документация может
фиксировать контракт или причину, которую правка нарушит. Если приступаешь к коду,
не открыв соответствующие документы, — задача начата неверно.

### 0.2. Все Markdown-документы хранятся в `docs/`

Вся проектная документация в формате `.md` лежит **только** в каталоге `docs/`.
Новый документ создаётся сразу там, а не в корне или рядом с кодом.

- Исключения — два корневых файла:
  - `CLAUDE.md` — остаётся в корне, потому что инструментарий (Claude Code)
    автоматически подхватывает его именно оттуда. Это точка входа в правила.
  - `README.md` — остаётся в корне по соглашению (его ожидают GitHub/npm). В нём
    только приветствие: что за игра, возможности, запуск сервера и как
    подключиться другому игроку. Структуры, карты модулей и архитектурных заметок
    в README быть не должно.
- Документация разбивается **по логике** на отдельные файлы в `docs/`, а не
  складывается в один: устройство (`architecture.md`), серверные модули
  (`server.md`), клиентские (`client.md`), контракт сообщений (`protocol.md`) и
  т. д. Новый крупный пласт знаний — новый файл, а не раздел в README.
- Ссылки между документами держим относительными и актуальными (из `docs/` на
  корневой `CLAUDE.md` — это `../CLAUDE.md`).

### 0.3. На каждую новую логику — обновлять `docs`

Любое изменение, добавляющее или меняющее **поведение** (новое действие, правило
игры, поле состояния, тип WebSocket-сообщения, модуль/сервис), сопровождается
обновлением документации в том же изменении (PR/коммите).

- Новый `type` сообщения → описать его (поля, кто вправе отправлять) в
  [docs/protocol.md](docs/protocol.md).
- Новое игровое правило или число-«магия» → отразить в документации правил.
- Новый модуль/сервис → добавить в карту модулей
  ([docs/server.md](docs/server.md) или [docs/client.md](docs/client.md)).
- Изменение архитектурного решения → обновить [docs/architecture.md](docs/architecture.md)
  и при необходимости соответствующий раздел здесь.

Правило простое: **код и документация меняются вместе**. PR с новой логикой, но
без обновления docs, считается незавершённым.

## 1. Правила комментирования

### 1.1. Каждая функция должна быть описана

Над **каждой** функцией (и сервером, и клиентом) ставится JSDoc-блок, который
отвечает на три вопроса: *что делает*, *что принимает*, *что возвращает*.

```js
/**
 * Назначает капитанов и раскидывает игроков по командам случайным образом.
 * Меняет объект room на месте (мутация), ничего не возвращает.
 *
 * @param {Room} room - комната, игроков которой нужно перемешать
 * @returns {void}
 */
function shuffleTeams(room) { /* ... */ }
```

Требования к JSDoc:

- Первая строка — краткое описание в одном предложении (что делает функция).
- Указывать `@param` для каждого аргумента с типом и смыслом.
- Указывать `@returns` (если функция мутирует состояние и ничего не
  возвращает — писать `@returns {void}` и пометку «мутирует …»).
- Для побочных эффектов (рассылка по WebSocket, запуск таймера, запись в лог)
  явно указывать их в описании — это критично для authoritative-сервера.

### 1.2. Комментировать логику, а не синтаксис

Комментарий объясняет **почему**, а не **что**. Хороший комментарий описывает
неочевидное решение, граничный случай или бизнес-правило игры.

```js
// ХОРОШО — объясняет правило игры и неочевидность:
// number === 0 трактуем как «безлимит»: даём большой счётчик попыток,
// чтобы команда могла открывать карты до первой ошибки.
room.guessesRemaining = number === 0 ? 99 : number + 1;

// ПЛОХО — пересказывает код:
// прибавляем единицу к number
room.guessesRemaining = number + 1;
```

### 1.3. Блоки логики разделяются заголовками-секциями

Внутри файла связанные функции группируются под баннер-комментарием, как уже
сделано в [server.js](server.js) (`// ---------- Таймер ----------`). После
разбиения на модули каждый такой блок становится отдельным файлом.

### 1.4. Обязательно комментировать

- **Игровые правила и числа-«магия»**: раскладка карт (`startCount`, `assassins`),
  бонусное время, размеры поля. Любое число, взятое из правил Codenames.
- **Граничные случаи и защита**: проверка выхода за пределы `public/`, валидация
  входящих сообщений, проверка прав (`hostId`, `role`, `team`).
- **Сетевые контракты**: формат каждого WebSocket-сообщения (`type` + поля).
- **Неочевидные UI-хаки**: например, почему таймер не входит в `structuralSig`
  (чтобы тик не сбрасывал фокус с поля ввода подсказки).

### 1.5. Документировать публичный контракт модуля

У каждого нового модуля сверху файла — блок-комментарий с назначением модуля и
перечислением экспортируемых функций.

```js
/**
 * @module services/gameEngine
 * Чистая игровая логика Codenames: старт партии, ходы, подсказки, проверка
 * победы. Не знает про WebSocket и HTTP — работает только с объектом Room.
 * Экспорт: startGame, giveClue, makeGuess, endTurn, checkWin.
 */
```

## 2. Архитектурные правила

### 2.1. Сервер — единственный источник правды (authoritative)

- Всё игровое состояние и таймеры живут **на сервере**. Клиент только отправляет
  намерения (`giveClue`, `guess`, …) и отрисовывает присланный `state`.
- Клиенту **никогда** не отправляются скрытые данные: цвета карт обычный агент
  получает только после открытия карты или в конце игры (см. `stateFor`).
- Любое новое действие должно валидироваться на сервере: фаза, права игрока
  (`hostId`/`role`/`team`), границы значений. Доверять клиенту нельзя.

### 2.2. Разделение слоёв (после модуляризации)

Зависимости направлены строго в одну сторону: **транспорт → приложение →
домен**. Доменный слой не знает о вышележащих.

```
transport (http, ws)   →  знает про сокеты и сообщения
        ↓
application (handlers) →  разбирает сообщения, проверяет права, вызывает домен
        ↓
domain/services        →  чистая логика игры, без I/O
        ↓
core (model, utils)    →  типы, константы, чистые помощники (shuffle, randomCode)
```

- **Доменные сервисы — чистые**: никаких `ws.send`, `setInterval`, `console`.
  Они принимают `room`/данные и меняют состояние. Рассылку и таймеры запускает
  вышележащий слой через колбэки/события.
- **Транспорт — тонкий**: парсинг сообщения, маршрутизация в handler, сериализация
  ответа. Игровых правил здесь нет.

### 2.3. Состояние и публичные данные разделены

- Внутренние поля комнаты (`_sockets`, `_interval`) помечаются префиксом `_` и
  **никогда** не попадают в `stateFor`. Сериализация для клиента — единственная
  точка, где формируется внешний контракт.
- При добавлении поля в `room` сознательно решай: оно внутреннее или часть
  состояния клиента. По умолчанию — внутреннее.

### 2.4. Единый формат сетевых сообщений

- Все сообщения — JSON с обязательным полем `type`.
- Входящие типы (клиент→сервер) и исходящие (сервер→клиент) перечислены в одном
  месте (см. план: `shared/messages.js` / константы). Никаких «магических строк»
  по коду — использовать константы.
- Любой новый `type` добавляется в общий список и документируется (поля, кто
  имеет право отправлять) — и в коде, и в `docs` (см. 0.3).

### 2.5. Управление ресурсами комнаты

- Таймеры (`_interval`) и сокеты (`_sockets`) — это ресурсы. На каждый запуск
  таймера должен существовать гарантированный `clearTimer`.
- Комната удаляется, когда не осталось подключённых игроков (`maybeCleanup`).
  Любой новый ресурс комнаты обязан освобождаться в этом же месте.

### 2.6. Клиент: рендер по состоянию, без дублирования истины

- Клиент не хранит игровую логику. Вся отрисовка — функция от `state`.
- Структурный рендер (поле, команды, кнопки) отделён от частого рендера
  (таймер) через сигнатуру `structuralSig`, чтобы не терять фокус/не моргать.
- DOM-доступ изолируется в одном слое (хелперы `$`, `$$` и будущие view-модули);
  бизнес-обработчики не лезут в DOM напрямую вперемешку с сетью.

### 2.7. Безопасность

- Статика отдаётся только из `public/`; защита от path traversal обязательна
  (проверка `filePath.startsWith(PUBLIC_DIR)`).
- Любой вывод пользовательских строк в DOM проходит через `escapeHtml`.
- Серверная валидация всех входных данных — обязательна (длина ника, диапазоны
  настроек, индекс карты, диапазон числа подсказки).

### 2.8. Конфигурация — через окружение и константы

- Изменяемые параметры (`PORT`, интервалы пинга, дефолтные настройки игры)
  выносятся в константы/`process.env`, не «зашиваются» по месту использования.

### 2.9. Зависимости и совместимость

- Серверные зависимости держим минимальными (сейчас только `ws`). Добавление
  новой зависимости обосновывается в PR.
- Клиент — без сборщика, нативные ES-модули в браузере. Серверная часть —
  CommonJS (`"type": "commonjs"`). Не смешивать в рамках одного слоя.

### 2.10. Стиль кода

- `'use strict';` во всех серверных файлах.
- Имена функций — глаголы (`startGame`, `giveClue`), данных — существительные.
- Одна функция — одна ответственность. Если функция делает «и логику, и
  рассылку, и лог» — это кандидат на разбиение по слоям (см. 2.2).
